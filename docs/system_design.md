# 系统设计说明

## 1. 系统目标

本系统用于通过微信公众号收集浇水上报信息，实现“自然语言上报 -> 待确认 -> 入库 -> 后台查询导出”的完整闭环。

## 2. 总体架构

系统采用分层设计：

1. 接入层：微信回调接口（验签、XML 解析、消息路由）
2. 业务层：消息处理、确认状态管理、记录落库
3. 数据层：SQLite 持久化（用户、地块、浇水记录、会话日志）
4. 管理层：记录后台与日志后台

## 3. 核心流程

### 3.1 上报流程

1. 用户向公众号发送文本或语音消息
2. 服务端校验 `signature/timestamp/nonce`
3. 解析消息内容并提取结构化字段
4. 生成待确认记录（`confirm_status=0`）
5. 返回确认提示（地块、农户、水量、日期、时间段）

### 3.2 确认流程

1. 用户回复 `1/确认`：将待确认记录更新为已确认（`confirm_status=1`）
2. 用户回复 `2/取消`：将待确认记录更新为已取消（`confirm_status=2`）
3. 若重复收到同一条上报文本：仅重发确认提示，不重复建记录

### 3.3 管理流程

1. 后台查询浇水记录（支持状态、日期、农户筛选）
2. 导出 CSV
3. 查看微信会话日志（入站、出站、错误）

## 4. 数据模型

### 4.1 用户表 `users`

- `openid`：微信 OpenID（唯一）
- `name`：组长昵称（优先真实昵称）

### 4.2 地块表 `plots`

- `plot_name`：地块名称
- `plot_code`：地块编码（唯一）
- `owner_name`：农户名称

### 4.3 浇水记录表 `watering_records`

- `user_id`、`plot_id`
- `volume`、`operation_date`、`start_time`、`end_time`
- `raw_input`：原始上报文本（含组长昵称和 OpenID）
- `confirm_status`：0 待确认 / 1 已确认 / 2 已取消

### 4.4 会话日志表 `wechat_message_logs`

- `openid`
- `direction`：`in` / `out`
- `msg_type`：消息类型
- `content`、`status`、`error`

## 5. 配置与部署

- 配置文件：`config.yaml`
- 脱敏模板：`config.yaml.example`
- 默认端口：`8000`
- 推荐运行：稳定模式（非 `--reload`）

## 6. 可观测性

- 应用日志：`logs/app.log`
- 聊天日志页面：`/api/v1/admin/log`
- 健康检查：`/api/v1/health`

## 7. 设计约束

- 微信被动回复需尽快返回，避免超时重试
- 数据不应在常规重启时被清空
- 敏感信息必须脱敏管理，不写入公开文档
