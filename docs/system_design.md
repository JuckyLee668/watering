# 微信智能浇水上报系统设计说明

## 1. 整体架构

系统按“接收 -> 解析 -> 确认 -> 存储”设计：

1. 微信消息网关：接收回调并完成签名验证
2. 大模型解析层：识别上报意图并提取字段
3. 本地状态层：保存待确认记录并设置超时（内存 + TTL）
4. 数据持久层：确认后写入 SQLite/MySQL/PostgreSQL
5. 管理查询层：后台统计与记录检索

## 2. 核心流程

### 2.1 上报流程

1. 用户发送自然语言上报消息
2. 后端验签并解析 XML 消息
3. 调用 LLM 返回结构化 JSON
4. 生成确认文本并写入本地状态机（默认 5 分钟）
5. 将确认消息返回给微信用户

### 2.2 确认流程

1. 用户回复“确认/1”或“取消/2”
2. 查询本地状态中的待确认数据
3. 若超时则提示重新上报
4. 若确认则写入 `watering_records` 并删除状态缓存

## 3. 数据模型

- `users`：用户基础信息
- `plots`：地块基础信息（由 CSV 同步）
- `watering_records`：浇水记录、原始输入、确认状态

地块主数据来源：`data/plots_sample.csv`

## 4. Prompt 约束建议

- 输出必须是 JSON
- 日期格式 `YYYY-MM-DD`
- 时间格式 `HH:MM`
- 对“今天/昨天/上午/下午”等时间表达做标准化
- 无关聊天返回 `{ "intent": "chat", "confidence": 0 }`
