# 微信智能浇水上报系统设计说明

## 1. 整体架构
系统按“接收 → 解析 → 确认 → 存储”设计：

1. 微信消息网关：接收回调消息并完成签名验证。
2. 大模型解析层：识别上报意图、提取地块/水量/时间。
3. Redis 状态层：保存待确认记录并设置超时。
4. 数据持久层：确认后落库到 MySQL/PostgreSQL。
5. 管理查询层：后台统计与记录检索。

## 2. 核心流程

### 2.1 上报流程
1. 用户发送自然语言上报消息。
2. 后端验签并解析 XML 消息。
3. 调用 LLM 返回结构化 JSON。
4. 生成确认文本并写入 Redis（默认 5 分钟）。
5. 将确认消息返回给微信用户。

### 2.2 确认流程
1. 用户回复“确认/1”或“取消/2”。
2. 查询 Redis 中待确认状态。
3. 若超时：提示重新上报。
4. 若确认：写入 watering_records 并删除 Redis 缓存。

## 3. 数据模型
当前数据库核心表：

- `users`：用户基础信息。
- `plots`：地块基础信息（由 CSV 同步生成）。
- `watering_records`：浇水记录、原始输入、确认状态。

地块主数据来源：`data/plots_sample.csv`。

详见 `scripts/init_database.sql`。

## 4. Prompt 约束建议

系统 Prompt 应明确：

- 输出格式必须是 JSON。
- 日期使用 `YYYY-MM-DD`，时间使用 `HH:MM`。
- 对“今天/昨天/上午/下午”等中文时间表达做标准化。
- 无关聊天场景返回 `{ "intent": "chat", "confidence": 0 }`。

## 5. 交互文案模板（建议）

```text
📋 浇水上报确认

地块：3号地
水量：50方
日期：2026-03-06
时间：14:00 - 16:00

请回复：
1/确认：提交记录
2/取消：放弃本次上报
直接重发：重新解析
```

## 6. 实施建议

- 先打通“微信回调 + Redis 状态机 + 数据入库”闭环。
- 再迭代“模糊地块纠错、异常检测、管理端导出”。
- 对低置信度结果追加二次提问，避免脏数据入库。
